# 设计输入规范

这个文件是给使用者看的。

如果你希望 AI 按你的设计要求开发，而不是按它自己的理解发挥，就尽量按这份规范提供信息。

## 目标

这份规范要解决一个常见问题：

- 用户觉得自己已经说清楚了
- 但 AI 实际上缺少关键信息
- 最后做出来的结果和预期不一致

所以这里不是增加负担，而是帮你把最关键的信息补全。

## 最少要给的信息

按设计开发时，至少提供这 5 类信息：

1. 目标页面或功能
2. 参考设计资料
3. 你最在意的视觉和交互点
4. 响应式要求
5. 验收标准

## 1. 目标页面或功能

至少说明：

- 是哪个页面、哪个功能、哪个模块
- 是新做，还是在现有基础上修改

例如：

- 修改登录页
- 新增个人中心资料页
- 调整首页顶部导航

## 2. 参考设计资料

你可以提供任意一种：

- 截图
- 设计图
- Figma 链接
- 线框图
- 文字描述

推荐优先级：

1. 截图或设计图
2. Figma 链接
3. 详细文字描述

如果只有文字描述，也可以，但要尽量具体。

## 3. 你最在意的点

不要只说“做漂亮一点”。

更有效的方式是明确你最在意什么，例如：

- 层级感
- 留白
- 按钮风格
- 表单布局
- 信息密度
- 品牌感
- 简洁还是厚重
- 更偏后台工具感还是营销页面感

如果你最在意的点不说，AI 很容易按它自己的审美做。

## 4. 响应式要求

至少说明：

- 只看桌面端
- 还是桌面端和移动端都要处理

如果你有更明确要求，也可以补充：

- 移动端优先
- 桌面端优先
- 平板也要兼容

## 5. 验收标准

至少说明你如何判断“做对了”。

例如：

- 页面结构和设计图一致
- 主要按钮位置和层级符合设计
- 表单在移动端不挤压
- 切换状态后交互正常
- 不要破坏现有登录流程

## 最好补充的信息

如果你能补这些信息，结果会更稳定：

- 文案是否已经确定
- 图片和图标是否已有素材
- 是否允许 AI 自行补默认文案
- 是否允许 AI 调整共享组件
- 这是页面私有需求，还是以后要复用
- 是否需要空状态、加载态、报错态

## 按设计开发时最容易漏掉的内容

这些信息用户最容易忘记，但很关键：

- 加载中长什么样
- 没有数据时长什么样
- 接口失败时怎么提示
- 按钮禁用时什么样
- 表单报错怎么显示
- 移动端是否要重新排版

如果你不说，AI 往往会默认处理，但不一定符合你的想法。

## 推荐输入格式

你可以直接复制下面模板：

```text
先阅读 agent_protocol/PROTOCOL.md。

我要按设计要求开发。

目标页面或功能：
……

当前状态：
是新做 / 在现有页面基础上修改

设计资料：
……

我最在意的点：
……

响应式要求：
桌面端 / 移动端 / 都要

状态要求：
是否需要 loading、empty、error、disabled

文案来源：
已提供 / 可由 AI 先补默认文案

验收标准：
……

如果信息还不够，请先指出缺什么，再开始实施。
```

## 如果你只有一张图

那也可以，建议这样发给 AI：

```text
先阅读 agent_protocol/PROTOCOL.md。

我只有一张参考图，请你先根据这张图判断：
1. 这是页面问题还是共享能力问题
2. 我还缺哪些信息
3. 先不要动代码，先把缺失信息列出来
```
