# 框架思路

这个文件是给使用者看的。

它不要求你懂代码，但它会帮助你理解这个仓库的设计思想。理解这些之后，你和 AI 协作会更稳定，需求也更容易一次说清。

## 这个框架想解决什么问题

很多项目在刚开始时开发很快，但做一段时间之后会越来越乱。

常见原因不是功能太多，而是这些问题：

- 页面里直接堆太多可复用逻辑
- 不同页面各自复制一套类似组件
- 主题、接口、状态处理分散在很多地方
- 改一个地方会连带影响很多别处
- AI 每次都可能按自己的理解新造一套结构

这个框架的目标，就是尽量避免这种情况。

它希望做到：

- 页面代码只负责页面
- 公共能力沉淀到共享层
- 不同 app 尽量复用同一套基础能力
- AI 每次工作时都有明确落点，不会越写越散

## 为什么分成 `apps` 和 `packages`

可以把这个仓库理解成两层：

## `apps/*`：最终应用

这里放的是你最后会看到、会使用的应用。

例如：

- 登录页
- 首页
- 后台页面
- 业务流程页面

这一层更接近“产品本身”。

它关心的是：

- 页面长什么样
- 路由怎么组织
- 用户点击之后发生什么
- 某个业务流程如何串起来

简单理解：

- `apps` 负责把东西“用起来”

## `packages/*`：共享能力

这里放的是多个页面、多个 app 都可能复用的能力。

例如：

- 通用 UI 组件
- 主题变量
- API 请求基础设施
- i18n、query、错误模型

这一层更接近“底座”。

它关心的是：

- 能不能复用
- 能不能统一维护
- 能不能被多个页面共享
- 改一次之后是不是大家一起受益

简单理解：

- `packages` 负责把东西“沉淀下来”

## 为什么不能在页面里随便复制公共实现

因为复制一次很快，但后面会越来越贵。

如果一个按钮、表单、请求逻辑在多个页面各写一份，后面会出现：

- 一个地方改了，另一个地方忘了改
- UI 风格越来越不一致
- Bug 修复要改很多份
- AI 以后更难判断哪份才是“真正标准实现”

所以这个框架的核心思想不是“先快写出来”，而是：

- 能只属于一个页面的，就放页面里
- 明显会复用的，就沉淀到共享层

## 为什么 AI 需要先判断“落点”

同一个需求，放错位置，后期成本会完全不同。

比如：

- 改登录页文案，这是页面问题，通常落在 `apps/web`
- 新增一套多个页面复用的表单组件，这是共享组件问题，通常落在 `packages/ui-components`
- 改全局主题颜色，这是主题问题，通常落在 `packages/ui-theme`
- 改统一请求错误处理，这是服务层问题，通常落在 `packages/services`

如果 AI 一上来不判断落点，直接改代码，最常见的后果就是：

- 本来应该共享的能力被写死在某个页面里
- 本来只是页面问题，却错误地改坏了底层公共能力
- 代码暂时能跑，但后面更难维护

## 这个框架的基本思路

你可以把它记成一句话：

先判断这次改动属于哪一层，再开始实现。

再展开一点就是：

1. 页面问题，优先放页面层
2. 公共能力，优先放共享层
3. 主题问题，放主题层
4. 服务问题，放服务层
5. 共享 UI 不在业务页面里随便复制

## 作为用户，你不需要学会代码

你不需要掌握具体目录结构，也不需要会写 TypeScript。

你只需要尽量说清三件事：

- 你想改什么
- 你希望结果变成什么样
- 这是单个页面的问题，还是一个会重复使用的能力

如果第三点你也判断不了，没有关系。

你可以直接对 AI 说：

```text
先阅读 AGENT_PROTOCOL/PROTOCOL.md。

我不确定这个需求应该落在哪一层。
请先根据当前框架设计判断是页面问题、共享组件问题、主题问题，还是服务问题，再开始回答或修改。
```

## 什么叫“单页面问题”

通常是这些：

- 某个页面布局要调整
- 某个页面增加一块业务内容
- 某个页面的交互流程要修改
- 某个页面接一个已有接口

这类需求通常优先落在 `apps/web`。

## 什么叫“共享能力问题”

通常是这些：

- 很多页面都要用的组件
- 很多页面都要复用的样式模式
- 统一的请求逻辑
- 统一的错误处理
- 全局主题变量

这类需求通常优先落在 `packages/*`。

## 什么时候你应该主动告诉 AI“不要升到共享层”

如果你的需求只是一次性页面调整，或者你明确知道这个东西只会在一个页面使用，可以直接说明：

```text
这是页面私有需求，先不要抽象成共享能力。
```

## 什么时候你应该主动告诉 AI“要做成共享能力”

如果你的需求明显会反复出现，可以直接说明：

```text
这个能力后面会在多个页面复用，请优先按共享能力设计。
```

## 你和 AI 协作时最有用的表达方式

相比“帮我改一下这个页面”，更有效的是：

- 我要改哪个页面
- 改成什么效果
- 这是页面私有需求，还是希望做成共享能力
- 我最终如何验收

## 最后一句

这个框架不是为了让开发变慢，而是为了让项目在不断增长时不失控。

你不需要先成为程序员，才可以使用它。

你只需要知道：

- 页面问题，优先想页面
- 公共能力，优先想共享
- 不确定时，让 AI 先判断落点
