# @workspace/services 协议

## 宪章

`@workspace/services` 是全仓库的全局服务层，负责沉淀跨应用可复用的“通用能力服务”。

当前核心能力：
- API 基础封装（`api/base` + `http-client`）
- i18n 初始化与约定（`i18n`）

## 强制规则

- 仅放“跨应用复用”的通用服务，禁止放页面/业务私有逻辑。
- 新服务必须通过稳定子路径导出（`package.json#exports`），禁止隐式内部引用。
- 破坏性变更必须保持向后兼容策略（保留旧导出或提供迁移路径）。
- 服务实现必须与 UI 层解耦，禁止依赖 `ui-core`、`ui-components` 或 app 页面代码。
- 服务层不得依赖 app 注入的请求 base URL 来补齐业务路径；对外请求路径必须在服务代码中显式完整定义。
- `group` 的语义仅用于区分目标服务域，不得把它设计成路径前缀分组。

## 扩展边界

- 可以新增的典型通用服务：
  - 认证与会话辅助
  - 文件上传/下载能力
  - URL 与路由参数工具
  - 国际化辅助工具
  - 通用请求重试、错误归一化、日志埋点适配
- 不应新增的内容：
  - 具体页面流程编排
  - 与单一业务域强绑定的逻辑
  - 只能在某个 app 使用且不可复用的临时代码

## AI 执行约束

- 新增服务前先确认是否已有同类能力，优先扩展现有模块，避免重复造轮子。
- 变更 `api/base`、`http-client`、`i18n` 这类基础能力时，必须评估对现有 app 的影响。
- 新增导出时同步更新 `package.json#exports`，并保证 TypeScript 类型可用。
- 协议与临时实现冲突时，以协议为准。
